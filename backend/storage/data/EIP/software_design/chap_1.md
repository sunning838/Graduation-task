# 1과목 소프트웨어 설계 및 요구사항 확인

## 1. 소프트웨어 생명 주기 (Software Life Cycle)
- **개념**: 소프트웨어를 개발하기 위해 정의하고 설계, 운용, 유지보수 등의 과정을 각 단계별로 나눈 것이다. 소프트웨어 프로세스 모형 또는 소프트웨어 공학 패러다임이라고도 한다.

### 1.1. 폭포수 모형 (Waterfall Model)
- 소프트웨어 개발도 이전 단계로 돌아갈 수 없다는 전제하에 각 단계를 확실히 매듭짓고 그 결과를 철저하게 검토하여 승인 과정을 거친 후에 다음 단계를 진행하는 선형 순차적 개발 방법론이다.
- 가장 오래되고 폭넓게 사용된 고전적 생명 주기 모형이다.
- 모형을 적용한 경험과 성공 사례가 많으며, 각 단계가 끝난 후에는 다음 단계를 수행하기 위한 결과물이 명확하게 산출되어야 한다.

### 1.2. 프로토타입 모형 (Prototype Model, 원형 모형)
- 사용자의 요구사항을 정확히 파악하기 위해 실제 개발될 소프트웨어에 대한 견본(시제)품을 만들어 최종 결과물을 예측하는 모형이다.
- 시제품은 의뢰자나 개발자 모두에게 공동의 참조 모델이 된다.

### 1.3. 나선형 모형 (Spiral Model, 점진적 모형)
- 보헴(Boehm)이 제안한 것으로, 폭포수 모형과 프로토타입 모형의 장점에 **위험 분석** 기능을 추가한 점진적 모형이다.
- 나선을 따라 돌듯이 여러 번의 소프트웨어 개발 과정을 거쳐 점진적으로 완벽한 최종 소프트웨어를 개발한다.
- 프로세스: 계획 -> 분석 -> 개발 -> 평가
- 소프트웨어를 개발하면서 발생할 수 있는 위험을 관리하고 최소화하는 것을 목적으로 한다.
- 점진적으로 개발 과정이 반복되므로 누락되거나 추가된 요구사항을 첨가할 수 있고, 정밀하며, 유지보수 과정이 필요 없다.

## 2. 애자일 (Agile) 방법론 <출제됨>
- **개념**: 절차보다는 사람이 중심이 되어 변화에 유연하고 신속하게 적응하면서 효율적으로 시스템을 개발할 수 있는 신속 적응적 경량 개발 방법론이다. 특정 개발 방법론이 아니라 고객과의 소통에 초점을 맞춘 방법론을 통칭한다.
- 고객의 요구사항 변화에 유연하게 대응할 수 있도록 일정한 주기를 반복하면서 개발과정을 진행하며 폭포수 모형과 대조적이다.
- 짧고 신속, 즉시 피드백, 유동적 개발.
- **유형**: XP, 린(Lean), 스크럼(SCRUM) 등.
- **애자일 4가지 핵심 가치**:
  1. 프로세스와 도구보다는 **개인과 상호작용**에 더 가치를 둔다.
  2. 방대한 문서보다는 **실행되는 SW**에 더 가치를 둔다.
  3. 계약 협상보다는 **고객과 협업**에 더 가치를 둔다.
  4. 계획을 따르기 보다는 **변화에 반응하는 것**에 더 가치를 둔다.

### 2.1. 스크럼 (Scrum)
- 팀이 중심이 되어 개발의 효율성을 높이는 기법이다. 매일 정해진 시간, 장소에서 짧은 시간의 개발을 하는 팀을 위한 프로젝트 관리 중심 개발론이다.
- 팀원 스스로가 스크럼 팀을 구성하고 개발 작업에 관한 모든 것을 스스로 해결할 수 있어야 한다.

**[스크럼 팀의 구성원과 역할]**
- **제품 책임자 (PO)**: 요구사항이 담긴 백로그를 작성하는 주체이다. 이해관계자들 중 개발될 제품에 대한 이해도가 높고, 요구사항을 책임지고 의사 결정할 사람으로 선정한다. 우선순위를 갱신한다. (백로그 관리)
- **스크럼 마스터 (SM)**: 스크럼 팀이 스크럼을 잘 수행할 수 있도록 객관적인 시각에서 조언을 해주는 가이드 역할을 수행한다. 일일 스크럼 회의를 주관하고 장애 요소를 처리한다.
- **개발팀 (DT)**: 제품 책임자(PO)와 스크럼 마스터(SM)를 제외한 모든 팀원으로 실제 제품 개발을 수행한다.

**[스크럼 개발 프로세스 및 주요 개념]**
- 프로세스 순서: 계획 -> 진행 -> 회의 -> 검토 -> 회고
- **제품 백로그**: 모든 요구사항(User Story)을 우선순위에 따라 나열.
- **스프린트 계획 회의**: 단기 일정 수립.
- **스프린트**: 실제 개발 작업 진행 (보통 2~4주).
- **일일 스크럼 회의 (스크럼 미팅)**: 매일 약 15분 진행 상황 점검. 
- **번 다운 차트 (소멸 차트, Burn-down Chart)**: 남은 작업 시간을 표시.
- **스프린트 검토 회의**: 완성 제품이 요구사항에 잘 부합되는지 테스팅.
- **스프린트 회고**: 규칙 준수 여부, 개선할 점 등을 확인하고 기록.

### 2.2. XP (eXtreme Programming)
- 수시로 발생하는 고객의 요구사항에 유연하게 대응하기 위해 고객의 참여와 개발 과정의 반복을 극대화하여 개발 생산성을 향상시키는 방법이다.
- 짧고 반복적인 개발 주기(1 ~ 3주의 반복(Iteration) 개발주기), 단순한 설계, 고객의 적극적인 참여를 통해 SW를 빠르게 개발한다.
- **5가지 핵심 가치**: 용기, 단순성, 의사소통, 피드백, 존중
- **XP 개발 프로세스**: 1. 릴리즈 계획 수립 -> 2. 이터레이션 (주기) -> 3. 승인 검사 -> 4. 소규모 릴리즈

**[XP의 12가지 기본원리 및 주요 실천 방법]**
1. **Pair Programming (짝 프로그래밍)**: 다른 사람과 함께 프로그래밍을 수행함으로써 개발에 대한 책임을 공동으로 나눠 갖는 환경을 조성한다.
2. **Collective Ownership (공동 코드 소유)**: 개발 코드에 대한 권한과 책임을 공동으로 소유한다.
3. **Test-Driven Development (테스트 주도 개발)**: 개발자가 실제 코드를 작성하기 전에 테스트 케이스를 먼저 작성하므로, 무엇을 해야 할지를 정확히 파악한다. 테스트가 지속적으로 진행될 수 있도록 자동화된 테스팅 도구를 사용한다.
4. **Whole Team (전체 팀)**: 고객 포함 개발에 참여하는 모든 구성원들은 각자 자신의 역할이 있고 그 역할에 대한 책임감을 가져야 한다.
5. **Continuous Integration (계속적인 통합)**: 모듈 단위로 나눠서 개발된 코드들은 하나의 작업이 마무리 될 때마다 지속적으로 통합된다.
6. **Refactoring (리팩토링)** <출제됨>: 프로그램 기능의 변경 없이 시스템을 단순화, 유연성 강화를 통해 재구성한다. (목적: 프로그램을 쉽게 이해하고 쉽게 수정하여 빠르게 개발할 수 있도록 하기 위함)
7. **Small Releases (소규모 릴리즈)**: 릴리즈 기간을 짧게 반복함으로써 고객의 요구 변화에 신속히 대응할 수 있다.
8. **계획 세우기**
9. **메타포어**
10. **간단한 디자인**
11. **40시간 작업**
12. **고객 상주**
13. **코드 표준** (원문에서는 12가지로 명시됨)

### 2.3. 린 (Lean)
- 도요타의 린 시스템 품질기법을 소프트웨어 개발 프로세스에 적용해서 낭비 요소를 제거하여 품질을 향상시킨 방법론.
- JIT(Just In Time), 칸반(Kanban) 보드 사용.
- **린(Lean)의 7가지 원칙**: 1. 낭비제거, 2. 품질 내재화, 3. 지식 창출, 4. 늦은 확정, 5. 빠른 인도, 6. 사람 존중, 7. 전체 최적화

## 3. 소프트웨어 개발 방법론 개요
- 소프트웨어 개발, 유지보수 등에 필요한 여러 가지 일들의 수행 방법과 이러한 일들을 효율적으로 수행하려는 과정에서 필요한 각종 기법 및 도구를 체계적으로 정리하여 표준화한 것이다.
- 소프트웨어 개발 방법론의 목적은 소프트웨어의 생산성과 품질 향상이다.

### 3.1. 구조적 방법론
- 정형화된 분석 절차에 따라 사용자의 요구사항을 파악하여 문서화하는 처리 중심의 방법론이다.
- 쉬운 이해 및 검증이 가능한 프로그램 코드를 생성하는 것이 목적이다.
- 복잡한 문제를 다루기 위해 분할과 정복 원리를 적용한다.
- **개발 절차**: 타당성 검토 -> 계획 -> 요구사항 -> 설계 -> 구현 -> 시험 -> 운용/유지보수

### 3.2. 정보공학 방법론
- 정보 시스템의 개발을 위해 계획, 분석, 설계, 구축에 정형화된 기법들을 상호 연관성 있게 통합 및 적용하는 자료 중심의 방법론이다.
- 정보 시스템 개발 주기를 이용하여 대규모 정보 시스템을 구축하는데 적합하다.
- **개발 절차**: 정보 전략 계획 수립 -> 업무 영역 분석 -> 업무 시스템 설계 -> 업무 시스템 구축

### 3.3. 컴포넌트 기반 방법론
- 기존의 시스템이나 소프트웨어를 구성하는 컴포넌트를 조합하여 하나의 새로운 애플리케이션을 만드는 방법론이다.
- 컴포넌트의 재사용이 가능하여 시간과 노력을 절감할 수 있다.
- 새로운 기능을 추가하는 것이 간단하여 확장성이 보장된다.
- 유지보수 비용을 최소화하고 생산성 및 품질을 향상시킬 수 있다.
- **개발 절차**: 개발 준비 -> 분석 -> 설계 -> 구현 -> 테스트 -> 전개 -> 인도

### 3.4. 제품 계열 방법론
- 특정 제품에 적용하고 싶은 공통된 기능을 정의하여 개발하는 방법론이다.
- 임베디드 소프트웨어를 만드는데 적합하다.
- 제품 계열 방법론은 영역공학과 응용공학으로 구분된다.

## 4. 객체지향 (Object-Oriented) 분석 및 설계
- 현실 세계의 개체를 기계의 부품처럼 하나의 객체로 만들어, 소프트웨어를 개발할 때 기계의 부품을 조립하듯이 객체들을 조립해서 필요한 소프트웨어를 구현하는 방법론이다.
- 구조적 기법의 문제점으로 인한 소프트웨어 위기의 해결책으로 채택되었다.
- **구성 요소**: 객체, 클래스, 메시지 등
- **개발 절차**: 요구 분석 -> 설계 -> 구현 -> 테스트 및 검증 -> 인도

### 4.1. 객체지향 기본 개념
- **객체 (Object)**: 데이터(속성)와 데이터를 처리하는 함수(메소드)를 묶어 놓은(캡슐화한) 하나의 소프트웨어 모듈. 독립적으로 식별 가능하며 상태는 변한다.
- **클래스 (Class)**: 공통된 속성과 연산을 갖는 객체의 집합으로, 객체의 일반적인 타입(Type). 데이터를 추상화하는 단위이며, 생성된 객체를 인스턴스라 함.
- **캡슐화 (Encapsulation)**: 데이터와 함수를 하나로 묶어 외부에서의 접근을 제한(정보 은닉). 외부 모듈의 변경으로 인한 파급 효과가 적고 재사용이 용이함.
- **상속 (Inheritance)**: 상위 클래스의 모든 속성과 연산을 하위 클래스가 물려받는 것.
- **다형성 (Polymorphism)**: 하나의 메시지에 대해 각각의 객체가 고유한 방법으로 응답할 수 있는 능력. (예: 오버로딩, 오버라이딩).
- **연관성 (Relationship)**: 객체 간의 상호 참조 관계. (is member of: 연관화, is instance of: 분류화, is part of: 집단화, is a: 일반화).

### 4.2. 객체지향 분석 방법론 종류
- **OMT [럼바우(Rumbaugh)] 방법**: 그래픽 표기법을 이용하여 소프트웨어 구성요소를 모델링하는 방법론. 분석 활동을 **객체 모델링** -> **동적 모델링** -> **기능 모델링** 순으로 수행.
  - **객체 모델링 (Object Modeling)**: 정보 모델링이라고도 함. ER 다이어그램을 만드는 과정까지의 모델링. **객체 다이어그램을 활용하여 표현**.
  - **동적 모델링 (Dynamic Modeling)**: 시간의 흐름에 따라 객체들 사이의 제어 흐름, 동작 순서 등의 동적인 행위를 표현. **상태 다이어그램을 활용하여 표현**.
  - **기능 모델링 (Functional Modeling)**: 프로세스들의 자료 흐름을 중심으로 처리 과정을 표현. **자료 흐름도(DFD)를 활용하여 표현**.
- **OOSE [야콥슨(Jacobson)] 방법**: 유스케이스에 의한 접근 방법으로 유스케이스를 모든 모델의 근간으로 활용하는 방법론. 분석, 설계, 구현 단계로 구성되며 기능적 요구사항 중심의 시스템이다.
- **OOD [부치(Booch)] 방법**: 설계 문서화를 강조하여 다이어그램 중심으로 개발하는 방법론. 미시적/거시적 개발 프로세스 모두 사용하며 분석과 설계의 분리가 불가능하다. 분석하는 데 이용된 객체 모델을 설계 시 적용.
- **Coad와 Yourdon 방법**: E-R 다이어그램을 사용하여 모델링.
- **Wirfs-Brock 방법**: 분석과 설계 간의 구분이 없음.

### 4.3. 객체지향 설계 원칙 (SOLID)
1. **단일 책임의 원칙 (SRP; Single Responsibility Principle)**
   - 하나의 클래스는 하나의 목적을 위해서 생성되며, 클래스가 제공하는 모든 서비스는 단 하나의 책임을 수행하는 데 집중되어 있어야 한다는 원칙.
2. **개방 폐쇄 원칙 (OCP; Open-Closed Principle)**
   - 소프트웨어의 구성요소(컴포넌트, 클래스, 모듈, 함수)는 확장에는 열려 있고, 코드를 변경하지 않고 기능 추가가 가능하도록 변경에는 닫혀있어야 한다는 원칙.
3. **리스코프 치환의 원칙 (LSP; Liskov Substitution Principle)**
   - 서브 타입(상속받은 자식/하위 클래스)은 어디서나 최소한 자신의 기반 타입(부모/상위 클래스)에서 가능한 행위를 수행하고 교체할 수 있어야 한다는 원칙.
4. **인터페이스 분리의 원칙 (ISP; Interface Segregation Principle)**
   - 한 클래스는 자신이 사용하지 않는 인터페이스는 구현(의존 관계)하지 말아야 한다는 원칙. 객체 설계 시 특정 기능에 대한 인터페이스는 그 기능과 상관없는 부분이 변해도 영향을 받지 않아야 한다.
5. **의존성 역전의 원칙 (DIP; Dependency Inversion Principle)**
   - 실제 사용 관계는 바뀌지 않으며, 추상성이 낮은 클래스보다 추상성이 높은 클래스(추상 매개)와 메시지를 주고받음으로써 관계를 최대한 느슨하게 만드는 원칙.

## 5. UML (Unified Modeling Language)
- 시스템 분석, 설계, 구현 등 개발 과정에서 시스템 개발자와 고객 또는 개발자 상호 간의 의사소통이 원활하게 이루어지도록 표준화한 대표적인 객체지향 모델링 언어이다.
- **UML의 구성 요소**: 사물, 관계, 다이어그램

### 5.1. 사물 (Things)
- 다이어그램 안에서 관계가 형성될 수 있는 대상들을 말한다.
- **사물의 종류**: 구조 사물, 행동 사물, 그룹 사물, 주해 사물

### 5.2. UML - 관계 (Relationships)
- 사물과 사물 사이의 연관성을 표현하는 것이다.
- **연관 관계 (Association)**: 2개 이상의 사물이 서로 연관되어 있는 관계. 사물 사이를 실선으로 연결. 방향성은 화살표로 표현하며 양방향 관계의 경우 화살표를 생략하고 실선으로만 연결한다. 다중도를 선 위에 표기한다.
- **집합 관계 (Aggregation)**: 하나의 사물이 다른 사물에 포함되어 있는 관계이다. 포함하는 쪽과 포함되는 쪽은 서로 독립적이다. 포함되는 쪽에서 포함하는 쪽으로 **속이 빈 마름모**를 연결하여 표현한다.
- **포함 관계 (Composition)**: 집합 관계의 특수한 형태로, 포함하는 사물의 변화가 포함되는 사물에게 영향을 미치는 관계이다. 포함하는 쪽과 포함되는 쪽은 서로 독립될 수 없고 생명주기를 함께한다. 포함되는 쪽에서 포함하는 쪽으로 **속이 채워진 마름모**를 연결하여 표현한다.
- **일반화 관계 (Generalization, 상속)**: 하나의 사물이 다른 사물에 비해 더 일반적이거나 구체적인 관계이다. 보다 일반적인 개념을 상위(부모), 구체적인 개념을 하위(자식)라고 부른다. 구체적(하위)인 사물에서 일반적(상위)인 사물 쪽으로 **속이 빈 화살표**를 연결하여 표현한다.
- **의존 관계 (Dependency)**: 연관 관계와 같이 사물 사이에 서로 연관은 있으나 소유 관계는 아니며, 사물의 변화가 다른 사물에도 영향을 미쳐 필요에 의해 짧은 시간 동안만 연관을 유지하는 관계이다. 영향을 주는 사물(이용자)이 영향을 받는 사물(제공자) 쪽으로 **점선 화살표**를 연결하여 표현한다.
- **실체화 관계 (Realization)**: 사물이 할 수 있거나 해야 하는 기능으로, 서로를 인터페이스로 그룹화 할 수 있는 관계이다. 사물에서 기능 쪽으로 **속이 빈 점선 화살표**를 연결하여 표현한다.

### 5.3. UML - 다이어그램 (Diagrams)
- 사물과 관계를 도형으로 표현한 것이다. 여러 관점에서 가시화한 뷰를 제공함으로써 의사소통에 도움을 준다.
- 정적 모델링에서는 주로 구조적 다이어그램을 사용하고, 동적 모델링에서는 주로 행위 다이어그램을 사용한다.

**[구조적(정적) 다이어그램의 종류]**
- **클래스 다이어그램**: 클래스와 클래스가 가지는 속성, 클래스 사이의 관계를 표현한다. (시스템 구조 파악 및 문서화)
- **객체 다이어그램**: 클래스에 속한 사물(객체)들, 즉 인스턴스를 특정 시점의 객체와 객체 사이의 관계로 표현한다. 럼바우 객체지향 분석 기법에서 객체 모델링에 활용된다.
- **컴포넌트 다이어그램**: 실제 구현 모듈인 컴포넌트 간의 관계나 컴포넌트 간의 인터페이스를 표현한다. (구현 단계에서 사용)
- **배치 다이어그램**: 결과물, 프로세스, 컴포넌트 등 물리적 요소들의 위치와 통신 경로를 표현한다. (구현 단계에서 사용)
- **복합체 구조 다이어그램**: 클래스나 컴포넌트가 복합 구조를 갖는 경우 그 내부 구조를 표현한다.
- **패키지 다이어그램** <출제됨>: 유스케이스나 클래스 등의 모델 요소들을 그룹화한 패키지들의 의존 관계를 표현한다.

**[행위적(동적) 다이어그램의 종류]**
- **유스케이스 다이어그램**: 사용자의 요구를 분석하는 것으로, 시스템, 액터, 유스케이스, 관계로 구성된다. 기능 모델링 작업에 사용한다.
- **시퀀스(Sequence) 다이어그램**: 상호 작용하는 시스템이나 객체들이 주고받는 메시지를 시간의 흐름에 따라 액터, 객체, 생명선, 실행 상자, 메시지로 표현한다.
- **커뮤니케이션 다이어그램**: 동작에 참여하는 객체들이 주고받는 메시지뿐만 아니라 객체들 간의 연관 관계까지 표현한다.
- **상태(State) 다이어그램**: 하나의 객체 자신이 속한 클래스의 상태 변화 혹은 다른 객체와의 상호 작용에 따라 상태가 어떻게 변화하는지를 표현한다. (럼바우 동적 모델링에 사용)
- **활동(Activity) 다이어그램**: 시스템이 어떤 기능을 수행하는지 기능 처리 로직이나 조건에 따른 처리의 흐름 순서에 따라 표현한다.
- **상호작용 개요 다이어그램**: 상호작용 다이어그램 간의 제어 흐름을 표현한다.
- **타이밍 다이어그램**: 객체 상태 변화와 시간 제약을 명시적으로 표현한다.

### 5.4. 스테레오 타입
- UML에서 표현하는 기본 기능 외에 추가적인 기능을 표현하는 것이다.
- `<< >>` (겹화살괄호/길러멧) 사이에 표현할 형태를 기술한다.
- **주요 형태**:
  - `<<include>>`: 연결된 다른 UML 요소에 대해 포함 관계에 있는 경우
  - `<<extend>>`: 연결된 다른 UML 요소에 대해 확장 관계에 있는 경우
  - `<<interface>>`: 인터페이스를 정의하는 경우
  - `<<exception>>`: 예외를 정의하는 경우
  - `<<constructor>>`: 생성자 역할을 수행하는 경우

## 6. 기능 모델링 / 정적 모델링 / 동적 모델링 상세

### 6.1. 기능 모델링 및 유스케이스 다이어그램
- **기능 모델링**: 개발될 시스템이 갖추어야 할 기능을 정리하여 전반적인 형태를 기능에 초점 맞춰 그림으로 표현 (유스케이스, 활동 다이어그램 등).
- **유스케이스 다이어그램**: 사용자와 다른 외부 시스템들이 개발될 시스템을 이용해 수행할 수 있는 기능을 사용자 관점에서 표현. 시스템의 범위를 파악하고 요구사항을 분석하는 도구.
- **구성 요소**: 
  - **시스템 / 시스템 범위**: 내부 유스케이스를 사각형으로 묶음.
  - **액터**: 외부요소. (주액터: 사람 등 이득을 얻는 대상 / 부액터: 외부 시스템, 기관 등 서비스 제공 목적)
  - **유스케이스**: 시스템이 액터에게 제공하는 서비스나 기능.
  - **관계**: 포함, 확장, 일반화 관계 등.

### 6.2. 활동 (Activity) 다이어그램 상세
- 사용자 관점에서 시스템이 수행하는 기능을 처리 흐름에 따라 순서대로 표현. 자료 흐름도와 유사.
- **구성 요소**:
  - **액션 / 액티비티**: 액션(더 분해할 수 없는 단일 작업), 액티비티(분리될 수 있는 작업)
  - **시작 / 종료 노드**: 시작 및 모든 흐름 종료.
  - **조건(판단) 노드**: 들어오는 제어 1개, 나가는 제어 여러 개.
  - **병합 노드**: 여러 경로의 흐름이 하나로 합쳐짐. (들어오는 제어 여러 개, 나가는 제어 1개)
  - **포크 노드**: 흐름이 분리되어 수행됨. (들어오는 흐름 1개, 나가는 흐름 여러 개)
  - **조인 노드**: 분리 수행되던 흐름이 다시 합쳐짐. (들어오는 흐름 여러 개, 나가는 흐름 1개)
  - **스웜 레인**: 액티비티 수행 담당 주체를 구분하는 선 (가로/세로 실선).

### 6.3. 정적 모델링 및 클래스 다이어그램 상세
- **정적 모델링**: 사용자가 요구한 기능을 구현하는데 필요한 자료들의 논리적인 구조를 객체/클래스로 추상화하여 표현.
- **클래스 다이어그램 구성 요소**:
  - **클래스**: 객체들이 갖는 속성과 오퍼레이션(동작/함수) 표현. 이름, 속성(상태/정보), 오퍼레이션 구획으로 나뉨.
  - **제약 조건**: 속성/오퍼레이션 제약. 중괄호 `{}` 이용.
  - **관계**: 연관, 집합, 포함, 일반화, 의존.
  - **연관 클래스**: 두 클래스의 연관 관계를 나타내는 선의 가운데로부터 점선을 이어 표시.

### 6.4. 동적 모델링 및 시퀀스 다이어그램 상세
- **동적 모델링**: 시스템 실행 시 구성 요소들 간의 상태 변화, 메시지 호출(오퍼레이션) 상호 작용 표현 (시퀀스, 커뮤니케이션, 상태 다이어그램).
- **시퀀스 다이어그램 구성 요소**:
  - **액터**: 서비스를 요청하는 외부 요소.
  - **객체**: 메시지 주체.
  - **생명선 (Lifeline)**: 객체가 존재하는 기간, 점선으로 표현.
  - **실행 상자**: 구동되고 있음을 표현.
  - **메시지**: 상호 작용 요소.
  - **객체 소멸**: 소멸(X) 표시, 더 이상 메모리에 존재 안 함.
  - **프레임**: 다이어그램을 묶어 표현.

### 6.5. 패키지 다이어그램 상세 <출제됨>
- 유스케이스나 클래스 등의 모델 요소들을 그룹화한 패키지 간의 의존 관계를 표현. 패키지는 또 다른 패키지의 요소가 될 수 있음. 대규모 시스템에서 종속성 파악 용이.
- **구성 요소**:
  - **패키지**: 단순 표기법(이름만 표현), 확장 표기법(요소까지 표현).
  - **객체**: 패키지에 포함될 유스케이스, 클래스, 인터페이스 등.
  - **의존 관계**: 점선 화살표 연결 (스테레오 타입 이용 가능).

## 7. 디자인 패턴 (Design Pattern) - GoF
- 모듈의 세부적인 구현 방안을 설계할 때 참조할 수 있는 전형적인 해결 방식.

### 7.1. 디자인 패턴 유형
1. **목적**
   - **생성**: 객체 인스턴스 생성에 관여, 클래스 정의와 객체 생성 방식을 구조화, 캡슐화를 수행하는 패턴.
   - **구조**: 더 큰 구조 형성 목적으로 클래스나 객체의 조합을 다루는 패턴.
   - **행위**: 클래스나 객체들이 상호 작용하는 방법과 역할 분담(책임 분배)을 다루는 패턴.
2. **범위**
   - **클래스**: 클래스 간 관련성(상속 관계를 다루는 패턴). 컴파일 타임에 정적으로 결정.
   - **객체**: 객체 간 관련성을 다루는 패턴. 런타임에 동적으로 결정.

### 7.2. 디자인 패턴 종류 상세
**[생성 패턴 (Creational)]**
- **Builder (빌더)**: 복잡한 인스턴스를 조립(건축)하여 만드는 구조. 객체를 생성하는 방법(과정)과 객체를 구현(표현)하는 방법을 분리함으로써 동일한 생성 절차에서 서로 다른 표현 결과를 만들 수 있다.
- **Prototype (프로토타입)**: 처음부터 일반적인 원형을 만들어 놓고, 그것을 복사(복제)한 후 필요한 부분만 수정하여 객체를 생성하는 패턴.
- **Factory Method (팩토리 메소드)**: 상위 클래스에서 객체를 생성하는 인터페이스를 정의하고, 하위(서브) 클래스에서 인스턴스를 생성하도록 하는 방식. 생성할 객체의 클래스를 국한하지 않음.
- **Abstract Factory (추상 팩토리)**: 구체적인 클래스에 의존하지 않고 서로 연관되거나 의존적인 객체들의 조합을 만드는 인터페이스를 제공하는 패턴. 동일 주제의 다른 팩토리를 묶음.
- **Singleton (싱글톤)**: 전역 변수를 사용하지 않고 한 클래스에서 인스턴스 객체를 하나만 생성하도록 제한하며, 생성된 객체를 어디에서든지 참조할 수 있도록 보장하는 패턴.

**[구조 패턴 (Structural)]**
- **Bridge (브리지)**: 기능의 클래스 계층과 구현의 클래스 계층을 연결하고, 구현부에서 추상 계층을 분리하여 추상화된 부분과 실제 구현 부분을 독립적으로 확장할 수 있는 패턴.
- **Decorator (데코레이터)**: 기존에 구현되어 있는 클래스에 필요한 기능을 덧붙여 나가는(추가해 나가는) 설계 패턴. 객체의 결합을 통해 기능을 동적으로 유연하게 확장.
- **Facade (퍼싸드)**: 복잡한 시스템에 대하여 단순한 상위(Wrapper) 인터페이스를 제공함으로써 결합도를 낮추어 시스템 구조 파악을 쉽게 하는 통합 패턴.
- **Flyweight (플라이웨이트)**: 다수의 객체로 생성될 경우 모두가 갖는 본질적인 요소를 클래스 화하여 공유함으로써(가상 인스턴스 제공) 메모리를 절약하고, '클래스의 경량화'를 목적으로 하는 패턴.
- **Proxy (프록시)**: '실체 객체에 대한 대리 객체'로 접근 이전에 필요한 행동을 취해 접근을 제어하거나, 실제 이용 시 할당하여 메모리를 아끼고, 실체 객체를 드러나지 않게 하여 정보은닉을 수행하는 패턴.
- **Composite (컴포지트)**: 객체들의 관계를 트리 구조로 구성하여 부분-전체 계층을 표현, 복합 객체와 단일 객체를 동일하게 취급(다루도록)하는 패턴.
- **Adapter (어댑터)**: 인터페이스가 호환되지 않는 클래스들을 함께 이용할 수 있도록, 기존에 생성된 클래스에 중간에서 맞춰주는 역할의 타 클래스 인터페이스를 기존 인터페이스에 덧씌우는(변환하는) 패턴.

**[행위 패턴 (Behavioral)]**
- **Mediator (중재자)**: 상호 간 통신이 복잡해지는 것을 막기 위해 중간에 통제하고 지시하는 중재자를 두어 통신 빈도수를 줄이고 상호작용을 캡슐화하여 결합도를 낮추는 패턴.
- **Interpreter (인터프리터)**: 언어의 다양한 해석, 구문을 나누고 분리된 구문의 해석을 맡는 클래스를 작성하여 문법 자체를 캡슐화해 표현/해석하는 패턴.
- **Iterator (반복자)**: 컬렉션 내부 구조를 노출시키지 않으면서도 집합체 안에 들어있는 모든 항목(원소)에 반복자를 사용하여 순차적으로 접근할 수 있는 패턴.
- **Template Method (템플릿 메소드)**: 전체 일을 수행하는 상위 작업 구조(골격)는 바꾸지 않으면서 일부분을 서브 클래스로 캡슐화해 특정 단계에서 수행하는 세부 내역을 하위 클래스에서 구체화/변경하는 패턴.
- **Observer (옵서버)**: 한 객체의 상태가 바뀌면 그 객체에 의존하는 다른 객체들에 연락이 가고 자동으로 내용이 연동/갱신되는 일대다(1:N) 의존성 패턴.
- **State (상태)**: 객체 상태를 캡슐화하여 클래스화함으로써 상태에 따라 다르게 행위 내용을 변경 처리할 수 있는 패턴.
- **Visitor (방문자)**: 각 클래스 데이터 구조로부터 처리 기능을 분리하여 별도 클래스로 만들고, 해당 메서드가 구조를 돌아다니며(방문하여) 특정 작업을 수행하도록 만드는 패턴.
- **Command (커맨드)**: 요구사항/실행될 기능을 캡슐화(로그 기록)하여 주어진 여러 기능을 실행할 수 있는 추상 클래스와 맞는 서브 클래스가 선택되어 실행되는 재사용성 높은 패턴.
- **Strategy (전략)**: 알고리즘 군을 정의하고 각각 캡슐화하여 동적으로 필요할 때 서로 알고리즘(행위)을 자유롭게 교환/변환해서 사용할 수 있게 하는 패턴.
- **Memento (메멘토)**: 객체의 특정 시점 상태 정보를 저장하여 객체를 이전 상태로 복구시켜야 하는 경우 '작업 취소(Undo)' 기능을 개발할 때 사용하는 패턴.
- **Chain of Responsibility (책임 연쇄)**: 한 요청을 2개 이상의 객체에서 고리를 따라 하드코딩된 연결이 아닌 동적 연결에 따라 다르게 책임이 넘어가며 처리되도록 하는 패턴.

## 8. 현행 시스템 파악 및 개발 기술 환경 정의

### 8.1. 현행 시스템 파악
- **1단계**: 시스템 구성 파악, 시스템 기능 파악, 시스템 인터페이스 파악.
- **2단계**: 아키텍처 구성 파악, 소프트웨어 구성 파악(제품명, 라이선스 등).
- **3단계**: 하드웨어 구성 파악(서버 사양, 이중화 여부), 네트워크 구성 파악.

### 8.2. 개발 기술 환경 파악 및 정의
- **운영체제 (OS)**: 컴퓨터 시스템 자원을 효율적으로 관리하며 사용자 환경을 제공하는 소프트웨어. 
  - 고려사항: 가용성, 성능, 기술 지원, 주변 기기, 구축 비용.
  - **PC 종류**: 윈도즈(Windows)[Microsoft, 유지/관리 비용 장점], 유닉스(UNIX)[IBM/HP/SUN, 대용량 안정성 엔터프라이즈급], 리눅스(Linux)[Linus Torvalds, 중/대규모, 높은 보안성]. 
    - ※ 리눅스 기반 시스템이 TCO(하드웨어 및 소프트웨어 소유 비용)가 가장 적게 소요된다.
  - **모바일 종류**: 안드로이드(Android)[Google, 리눅스 위 구동, 자바/코틀린 런타임 라이브러리 제공 소프트웨어 스택], iOS[Apple, 높은 보안성과 고성능].
- **데이터베이스 관리 시스템 (DBMS)**: 사용자와 데이터베이스 사이에서 요구에 따라 정보 생성 및 DB 관리. 
  - 고려사항: 성능, 기술 지원, 상호 호환성, 구축 비용.
- **웹 애플리케이션 서버 (WAS)**: 사용자의 요구에 따라 변하는 동적인 콘텐츠를 처리하기 위해 사용되는 미들웨어. 
  - 고려사항: 가용성, 성능, 기술 지원, 구축 비용.
- **오픈 소스**: 누구나 별다른 제한 없이 사용할 수 있도록 소스 코드를 공개한 소프트웨어. 
  - 고려사항: 라이선스의 종류, 사용자 수, 기술의 지속 가능성.

## 9. 요구사항 (Requirements)
- **개념**: 소프트웨어가 어떤 문제를 해결하기 위해 제공하는 서비스에 대한 설명과 정상적으로 운영되는데 필요한 제약조건.

### 9.1. 요구사항의 분류 및 유형
- **기능(적) 요구사항** <출제됨>: 시스템이 제공하는 기능, 서비스에 대한 요구사항. 시스템이 무엇을 하는지 어떤 기능을 하는지 관련됨. (입출력, 데이터 저장, 연산 수행 등 반드시 수행해야 하는 기능).
  - 예시: [상품의 결제수단은 신용카드, 무통장 입금, 포인트 결제가 가능해야 함.]
- **비기능(적) 요구사항** <출제됨>: 성능, 보안, 품질, 인터페이스, 장비 구성, 데이터 구축, 테스트, 프로젝트 관리 등 시스템 구축 제약사항이나 기능 이외에 관한 요구사항.
  - 예시: [특정 함수의 호출시간은 3초를 넘지 않아야 함, 시스템은 하루 24시간 가동되어야 하며 가동률 99.5%를 만족해야 함.]
- **사용자 요구사항**: 사용자 관점에서 본 시스템이 제공해야 할 요구사항.
- **시스템 요구사항**: 개발자 관점에서 본 시스템 전체가 사용자와 다른 시스템에 제공해야 할 요구사항.
- **요구공학**: 무엇을 개발해야 하는지 요구사항을 정의하고, 분석 및 관리하는 프로세스를 연구하는 학문이다.

### 9.2. 요구사항 개발 프로세스
개발 대상에 대한 요구사항을 체계적으로 도출, 분석, 명세, 확인 및 검증하는 일련의 구조화된 활동이다.
1. **요구사항 도출 (Elicitation)**: 관련자들이 의견을 교환하여 요구사항을 수집하고 식별하는 과정. 주요 기법으로 청취와 인터뷰, 설문, 브레인스토밍, 워크샵, 프로토타이핑, 유스케이스가 있다.
2. **요구사항 분석 (Analysis)**: 사용자의 요구사항 중 명확하지 않거나 모호하여 이해되지 않는 부분을 발견하고 이를 걸러내는 첫 실제적인 단계 과정이다. 
3. **요구사항 명세 (Specification)**: 분석된 요구사항을 바탕으로 정형/비정형 기법을 활용해 모델을 작성하고 문서화하는 것을 의미한다.
   - 정형 명세 기법: 수학적 원리/기호 표기법. 완전성 검증 가능하나 어렵다. (VDM, Z, Petri-net, CSP 등)
   - 비정형 명세 기법: 상태/기능/객체 중심. 자연어나 다이어그램 서술. 소통 용이하나 해석이 달라질 수 있음. (FSM, Decision Table, ER 등)
4. **요구사항 확인 (Validation)**: 요구사항 명세서가 정확하고 완전하게 작성되었는지 자원 할당 전 검토(검증)하는 활동.

### 9.3. 요구사항 확인 및 검증 단계 주요 기법 (정형 기술 검토 활용)
1. **동료 검토 (Peer Review)**: 2~3명이 진행하는 리뷰 형태. 요구사항 명세서 작성자가 설명하고 이해관계자들이 설명을 들으면서 결함을 발견하는 검토 방법.
2. **워크 스루 (Walk Through)**: 오류를 조기에 검출하는 데 목적이 있는 비공식적 검토 방법. 검토 자료를 회의 전에 배포해서 사전검토한 후 짧은 시간 동안 회의를 진행해 오류 검출/문서화.
3. **인스펙션 (Inspection)**: 저작자 외의 다른 전문가 또는 팀이 검사하여 오류를 찾아내는 공식적인 검토 방법. 절차는 계획 -> 사전 교육 -> 준비 -> 인스펙션 회의 -> 수정 -> 후속 조치 순.

### 9.4. 구조적 분석 기법 및 분석 도구
- 자료의 흐름과 처리를 중심으로 하는 분석 방법.
- **자료 흐름도 (DFD; Data Flow Diagram, 버블 차트)**: 자료의 흐름, 변환, 기능을 도형 중심으로 기술.
  - 기호: 프로세스/기능/변환(원), 자료 흐름(화살표), 자료 저장소(평행선), 단말/외부개체(사각형).
- **자료 사전 (DD; Data Dictionary)**: DFD에 있는 자료를 더 자세히 정의한 메타 데이터.
  - 기호: `=`(정의), `+`(연결), `( )`(생략 가능), `[ | ]`(선택), `{ }`(반복), `* *`(주석).
- **기타 도구**: 소단위 명세서(Mini-Spec.), 개체 관계도(ERD), 상태 전이도(STD), 제어 명세서.

### 9.5. 요구사항 분석 CASE(자동화 도구)와 HIPO
- **요구사항 분석용 CASE**: 자동으로 요구사항 분석 명세서를 기록하는 도구.
  - **SADT**: SoftTech사 개발. 시스템/요구사항 분석 및 설계 도구로 블록 다이어그램 채택.
  - **SREM (RSL/REVS)**: TRW사 개발. 실시간 처리 소프트웨어 시스템 명확 기술용.
  - **PSL/PSA**: 미시간 대학 개발.
  - **TAGS**: 시스템 공학 방법 응용을 위한 통합 자동화 도구.
- **HIPO (Hierarchy Input Process Output)**: 시스템 실행 과정인 입력, 처리, 출력의 기능을 표현하는 기법. 하향식 개발을 지원하며 기능과 자료 의존 관계를 동시 표현.
  - 종류: 가시적 도표, 총체적 도표, 세부적 도표.

## 10. 소프트웨어 아키텍처 및 설계, 인터페이스
- **상위 설계**: 아키텍처 설계, 예비 설계 (시스템 구조, DB, 인터페이스 설계).
- **하위 설계**: 모듈 설계, 상세 설계 (컴포넌트, 자료 구조, 알고리즘 설계).
- **설계 기본 원리**:
  - **단계적 분해 (Stepwise Refinement)**: 상위 개념부터 하위 개념으로 구체화하는 하향식 분할 기법.
  - **정보 은닉 (Information Hiding)**: 내부 절차를 감추어 다른 모듈이 접근하지 못하도록 함.
  - **모듈화 (Modularity)**: 기능들을 모듈 단위로 나누어 유지 관리 향상.
  - **추상화 (Abstraction)**: 포괄적 개념 설계 후 세분화.

### 10.1. 모듈의 결합도와 응집도
- 소프트웨어 품질을 높이려면 **결합도는 약하게, 응집도는 강하게** 설계해야 한다. 팬인(Fan-In)은 높게, 팬아웃(Fan-Out)은 낮게 설계한다.
- **결합도 (Coupling)** : 강함(나쁨) -> 약함(좋음)
  - 내용 (Content) -> 공통 (Common) -> 외부 (External) -> 제어 (Control) -> 스탬프 (Stamp, 배열/자료구조) -> 자료 (Data, 단순 자료요소)
- **응집도 (Cohesion)** : 약함(나쁨) -> 강함(좋음)
  - 우연적 (Coincidental) -> 논리적 (Logical) -> 시간적 (Temporal) -> 절차적 (Procedural) -> 교환적/통신적 (Communication) -> 순차적 (Sequential) -> 기능적 (Functional)

### 10.2. 아키텍처 패턴
- **파이프-필터 패턴 (Pipe-Filter)**: 데이터 스트림 절차 각 단계를 필터로 캡슐화하고 파이프 전송. (예: UNIX 쉘). 재사용/확장 용이.
- **모델-뷰-컨트롤러 패턴 (MVC)**: Model(기능/데이터), View(화면 표시), Controller(요청 처리/제어).
- 기타: 마스터-슬레이브, 브로커, 피어-투-피어, 이벤트-버스, 블랙보드, 인터프리터.

### 10.3. 사용자 인터페이스 (UI) 설계 도구 및 기타
- **UI 설계 지침**: 사용자 중심, 사용성, 심미성, 오류 해결, 직관성, 유효성, 학습성, 유연성.
- **UI 설계 도구**: 
  - **와이어프레임**: 초기 뼈대 설계.
  - **목업 (Mockup)**: 실제 화면 유사 정적 모형.
  - **스토리보드**: 와이어프레임+설명 흐름, 최종 지침서.
  - **프로토타입**: 실제 구현처럼 인터랙션 동적 모형.
- **코드(Code) 종류**: 순차(일련번호), 블록(구분), 10진, 그룹 분류, 연상(약호), 표의 숫자(물리적 수치, 유효 숫자), 합성.

## 11. S/W 공학 발전적 추세 및 프레임워크

### 11.1. 소프트웨어 재사용 및 재공학
- **소프트웨어 재사용**: 인정받은 소프트웨어 부품을 재사용.
  - 합성 중심 (블록 구성 방법): 부품/블록을 끼워 맞춰 완성.
  - 생성 중심 (패턴 구성 방법): 추상화 명세를 구체화하여 완성.
- **소프트웨어 재공학**: 기존 시스템 이용하여 새로운 기능 추가, 성능 향상.
  - 이점: 품질 향상, 생산성 증가, 수명 연장, 오류 감소.
- **CASE (Computer Aided Software Engineering)**: 소프트웨어 개발 생명 주기 과정 전체/일부를 컴퓨터 자동화 도구로 지원. 다양한 모형/그래픽 지원.

### 11.2. 소프트웨어 개발 표준
- **ISO/IEC 12207**: ISO 표준 S/W 생명 주기 프로세스.
  - 기본 (획득/공급/개발/운영/유지보수), 지원 (품질보증/문서화 등), 조직 (관리/개선).
- **CMMI**: 소프트웨어 프로세스 성숙도 5단계 모델.
  - 1.초기 -> 2.관리 -> 3.정의(표준화) -> 4.정량적 관리 -> 5.최적화(지속적 개선).
- **SPICE**: 프로세스 평가 및 개선 국제 표준.
  - 프로세스 범주: 고객-공급자(10), 공학(9), 지원(8), 관리(4), 조직(9).
  - 수행 능력 단계: 1.불완전 -> 2.수행 -> 3.관리 -> 4.확립 -> 5.예측 -> 6.최적화.

### 11.3. 소프트웨어 개발 프레임워크
- 공통 사용 구성 요소와 아키텍처를 반제품 형태로 제공하는 시스템. 사업자 종속성 해소, 예외/트랜잭션/DB/로깅 등 지원.
- **특성**: 모듈화, 재사용성, 확장성, 제어의 역흐름.
- **종류**:
  - **스프링 프레임워크**: 자바 플랫폼 오픈 소스 경량형 애플리케이션 프레임워크. 전자정부 표준 기반 기술.
  - **전자정부 프레임워크**: 대한민국 공공부문 구축 프레임워크. 특정 업체 종속 배제.
  - **닷넷 프레임워크**: Windows 프로그램 환경. CLR 가상머신 상 작동.

## 12. 비용 산정 기법 및 프로젝트 일정 계획

### 12.1. 비용 산정 기법 - 하향식
- 과거 유사 경험과 전문 지식 회의를 통한 비과학적/전체 집계 세분화 기법.
- **전문가 감정 기법**: 2명 이상의 전문가 산정. 신속하나 개인적/주관적 편견 가능.
- **델파이 기법**: 전문가 편견 보완을 위해 1명의 조정자와 여러 전문가로 구성하여 의견을 종합.

### 12.2. 비용 산정 기법 - 상향식
- 세부적인 작업 단위별로 비용을 산정 후 집계.
- **LOC (원시 코드 라인 수) 기법** <출제됨>: 원시 코드 수 비관/낙관/기대치로 예측치 구함. 가장 많이 사용됨.
  - 산정 공식 (예측치) = `(낙관치 + 4×기대치 + 비관치) / 6`
  - 노력 = `개발 기간 × 투입 인원` = `LOC / 월평균 생산 코드 라인 수`
  - 기타 산출: 개발 비용, 개발 기간, 생산성.
- **개발 단계별 인월수 기법**: LOC 보완, 생명 주기 각 단계별 산정으로 더 정확함.

### 12.3. 수학적 산정 기법 (경험적 추정 모형)
- 상향식 기법 중 하나로 과거 프로젝트 기반 공식 유도, 비용 산정 자동화.
- **COCOMO 모형**: LOC 기반 비용/노력 산정.
  - 개발 유형: **조직형**(5만 이하, 응용S/W), **반분리형**(30만 이하, 유틸리티), **내장형**(30만 이상 초대형, 실시간 제어).
  - 종류: 기본형, 중간형(4가지 특성 반영), 발전형(공정별 상세).
- **Putnam 모형**: 생명 주기 전 과정 노력 분포 예상. RayLeigh-Norden 곡선 기초. 대형 프로젝트 사용 (개발 기간 늘면 노력 감소).
- **기능 점수 (FP) 모형**: 기능 증대 요인(자료 입력, 정보 출력, 명령어, 데이터 파일, 인터페이스)에 가중치를 부여 합산.
- **자동화 추정 도구**: SLIM (Putnam 기초), ESTIMACS (기능 점수 기초).
- **Man Month 모형**: `Man Month = LoC / 프로그래머 월간 생산성`, `프로젝트 기간 = Man Month / 프로젝트 인력`.

### 12.4. 프로젝트 일정 계획
- 소작업 분배 및 순서/일정 정하기.
- **PERT**: 소요 기간 예측이 어려운 프로젝트. 네트워크 노드와 화살표(간선)에 낙관/기대/비관치 표시. (결정 경로, 경계 시간 등 파악). 예측치 공식 `(비관+4×기대+낙관)/6`.
- **CPM**: 작업 나열 및 소요 기간 예측 기법. 전후 의존 관계를 노드(원형=작업, 박스=이정표)와 간선 화살표로 나타냄.
- **간트 차트 (시간선 차트)**: 작업 일정을 수평 막대 도표로 표시. 중간 목표/예산 관리, 이정표/산출물 파악 가능. 자원 배치에 유용.
