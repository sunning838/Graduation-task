# 1과목 소프트웨어 설계

## 1. 소프트웨어 생명 주기 (Software Life Cycle)
- **개념**: 소프트웨어를 개발하기 위해 정의하고 운용, 유지보수 등의 과정을 각 단계별로 나눈 것이다. 소프트웨어 프로세스 모형 또는 소프트웨어 공학 패러다임이라고도 한다.

### 폭포수 모형 (Waterfall Model)
- 소프트웨어 개발도 이전 단계로 돌아갈 수 없다는 전제하에 각 단계를 확실히 매듭짓고 검토하여 승인 과정을 거친 후에 다음 단계를 진행하는 선형 순차적 모형이다.
- 가장 오래되고 폭넓게 사용된 고전적 생명 주기 모형이다.
- 모형을 적용한 경험과 성공 사례가 많으며, 각 단계가 끝난 후에는 다음 단계를 수행하기 위한 결과물이 명확하게 산출되어야 한다.

### 프로토타입 모형 (Prototype Model, 원형 모형)
- 사용자의 요구사항을 정확히 파악하기 위해 실제 개발될 소프트웨어에 대한 견본(시제)품을 만들어 최종 결과물을 예측하는 모형이다.
- 시제품은 의뢰자나 개발자 모두에게 공동의 참조 모델이 된다.

### 나선형 모형 (Spiral Model, 점진적 모형)
- 보헴(Boehm)이 제안한 것으로, 폭포수 모형과 프로토타입 모형의 장점에 **위험 분석** 기능을 추가한 점진적 모형이다.
- 나선을 따라 돌듯이 여러 번의 개발 과정을 거쳐 점진적으로 완벽한 최종 소프트웨어를 개발한다.
- 소프트웨어를 개발하면서 발생할 수 있는 위험을 관리하고 최소화하는 것을 목적으로 한다.
- 점진적으로 개발 과정이 반복되므로 누락되거나 추가된 요구사항을 첨가할 수 있고, 정밀하며, 유지보수 과정이 필요 없다.

## 2. 애자일 모형 (Agile Model)
- **개념**: 고객의 요구사항 변화에 유연하게 대응할 수 있도록 일정한 주기를 반복하면서 개발과정을 진행하는 방법론이다. 특정 개발 방법론이 아니라 고객과의 소통에 초점을 맞춘 방법론을 통칭한다.
- **애자일 4가지 핵심 가치**:
  1. 프로세스와 도구보다는 **개인과 상호작용**에 더 가치를 둔다.
  2. 방대한 문서보다는 **실행되는 SW**에 더 가치를 둔다.
  3. 계약 협상보다는 **고객과 협업**에 더 가치를 둔다.
  4. 계획을 따르기 보다는 **변화에 반응하는 것**에 더 가치를 둔다.

### 스크럼 (Scrum)
- 팀이 중심이 되어 개발의 효율성을 높이는 애자일 방법론이다. 팀원 스스로가 스크럼 팀을 구성하고 개발 작업에 관한 모든 것을 스스로 해결할 수 있어야 한다.
- **제품 책임자 (PO)**: 요구사항을 책임지고 의사 결정할 사람. 요구사항을 작성하고 우선순위를 갱신한다. (백로그 관리)
- **스크럼 마스터 (SM)**: 스크럼 팀이 스크럼을 잘 수행할 수 있도록 객관적인 시각에서 조언을 해주는 가이드 역할. 일일 스크럼 회의를 주관하고 장애 요소를 처리한다.
- **개발팀 (DT)**: PO와 SM을 제외한 모든 팀원으로 실제 개발을 수행한다.
- **스크럼 개발 프로세스**: 
  - **제품 백로그**: 모든 요구사항(User Story)을 우선순위에 따라 나열.
  - **스프린트 계획 회의**: 단기 일정 수립.
  - **스프린트**: 실제 개발 작업 진행 (보통 2~4주).
  - **일일 스크럼 회의**: 매일 약 15분 진행 상황 점검. 남은 작업 시간은 소멸 차트(Burn-down Chart)에 표시.
  - **스프린트 검토 회의**: 완성 제품이 요구사항에 잘 부합되는지 테스팅.
  - **스프린트 회고**: 규칙 준수 여부, 개선할 점 등을 확인하고 기록.

### XP (eXtreme Programming)
- 수시로 발생하는 고객의 요구사항에 유연하게 대응하기 위해 고객의 참여와 개발 과정의 반복을 극대화하여 생산성을 향상시키는 방법이다.
- 짧고 반복적인 개발 주기, 단순한 설계, 고객의 적극적인 참여를 통해 SW를 빠르게 개발한다.
- **5가지 핵심 가치**: 용기, 단순성, 의사소통, 피드백, 존중.
- **XP의 주요 실천 방법 (Practice)**:
  - **Pair Programming(짝 프로그래밍)**: 다른 사람과 함께 개발하여 책임을 공동으로 나눔.
  - **Collective Ownership(공동 코드 소유)**: 개발 코드에 대한 권한과 책임을 공동 소유.
  - **Test-Driven Development(TDD, 테스트 주도 개발)**: 실제 코드 작성 전 테스트 케이스를 먼저 작성.
  - **Whole Team(전체 팀)**: 고객 포함 모든 구성원들이 각자 역할과 책임을 가짐.
  - **Continuous Integration(계속적인 통합)**: 모듈 단위로 지속적 통합.
  - **Design Improvement / Refactoring(리팩토링)**: 기능 변경 없이 단순화, 유연성 강화를 통해 시스템을 재구성.
  - **Small Releases(소규모 릴리즈)**: 릴리즈 기간을 짧게 반복.

## 3. 현행 시스템 파악
- **1단계**: 시스템 구성 파악, 시스템 기능 파악, 시스템 인터페이스 파악.
- **2단계**: 아키텍처 구성 파악, 소프트웨어 구성 파악(제품명, 라이선스 등).
- **3단계**: 하드웨어 구성 파악(서버 사양, 이중화 여부), 네트워크 구성 파악.

## 4. 요구사항 (Requirements)
- **개념**: 소프트웨어가 어떤 문제를 해결하기 위해 제공하는 서비스에 대한 설명과 제약조건.
- **기능 요구사항**: 시스템이 무엇을 하는지, 어떤 기능을 하는지에 대한 사항. (입출력, 데이터 저장, 연산 수행 등 시스템이 반드시 수행해야 하는 기능).
- **비기능 요구사항**: 성능, 보안, 품질, 인터페이스, 장비 구성, 데이터 구축, 테스트, 제약사항, 프로젝트 관리 등에 대한 요구사항.

### 요구사항 개발 프로세스
- **도출 (Elicitation)**: 요구사항 수집. (인터뷰, 설문, 브레인스토밍, 워크샵, 유스케이스 등)
- **분석 (Analysis)**: 요구사항 중 명확하지 않거나 모호한 부분을 발견하고 걸러내는 과정. (DFD, DD, ERD 등 사용)
- **명세 (Specification)**: 분석된 요구사항을 바탕으로 모델을 작성하고 문서화.
- **확인 (Validation)**: 요구사항 명세서가 정확하고 완전하게 작성되었는지 검토(검증)하는 활동.

### 요구사항 명세 기법
- **정형 명세 기법**: 수학적 원리/모델 기반. 수학적 기호와 정형화된 표기법 사용. 완전성 검증이 가능하지만 사용자가 이해하기 어려움. (VDM, Z, Petri-net, CSP 등)
- **비정형 명세 기법**: 상태/기능/객체 중심. 자연어 기반 서술이나 다이어그램 작성. 의사소통이 용이하지만 해석이 달라질 수 있음. (FSM, Decision Table, ER모델링 등)

### 요구사항 분석 도구
- **자료 흐름도 (DFD; Data Flow Diagram)**: 자료의 흐름 및 변환 과정과 기능을 도형 중심으로 기술. (버블 차트). 
  - 프로세스(원), 자료 흐름(화살표), 자료 저장소(평행선), 단말(사각형)
- **자료 사전 (DD; Data Dictionary)**: 자료 흐름도에 있는 자료를 정의한 메타 데이터. 
  - `=`(정의), `+`(연결), `( )`(생략 가능), `[ | ]`(선택), `{ }`(반복), `* *`(주석)
- **요구사항 분석용 CASE 도구**:
  - **SADT**: SoftTech사 개발. 구조적 분석 및 설계 도구.
  - **SREM (RSL/REVS)**: TRW사 개발. 우주 국방 시스템 등 실시간 처리용.
  - **PSL/PSA**: 미시간 대학 개발.
  - **TAGS**: 시스템 공학 방법 응용을 위한 통합 자동화 도구.
- **HIPO (Hierarchy Input Process Output)**: 시스템 실행 과정인 입력, 처리, 출력을 나타내는 문서화 도구. 하향식 소프트웨어 개발을 지원하며 가시적 도표, 총체적 도표, 세부적 도표로 구성됨.

## 5. 객체지향 (Object-Oriented) 분석 및 설계
- **객체 (Object)**: 데이터(속성)와 데이터를 처리하는 함수(메소드)를 묶어 놓은(캡슐화한) 하나의 소프트웨어 모듈. 독립적으로 식별 가능하며 상태는 변한다.
- **클래스 (Class)**: 공통된 속성과 연산을 갖는 객체의 집합으로, 객체의 일반적인 타입(Type). 데이터를 추상화하는 단위이며, 생성된 객체를 인스턴스라 함.
- **캡슐화 (Encapsulation)**: 데이터와 함수를 하나로 묶어 외부에서의 접근을 제한(정보 은닉). 외부 모듈의 변경으로 인한 파급 효과가 적고 재사용이 용이함.
- **상속 (Inheritance)**: 상위 클래스의 모든 속성과 연산을 하위 클래스가 물려받는 것.
- **다형성 (Polymorphism)**: 하나의 메시지에 대해 각각의 객체가 고유한 방법으로 응답할 수 있는 능력. (예: 오버로딩, 오버라이딩).
- **연관성 (Relationship)**: 객체 간의 상호 참조 관계. (is member of: 연관화, is instance of: 분류화, is part of: 집단화, is a: 일반화).

### 객체지향 분석 방법론
- **럼바우 (Rumbaugh) 방법 (OMT)**: 분석 활동을 **객체 모델링**(ER다이어그램) -> **동적 모델링**(상태 다이어그램) -> **기능 모델링**(자료 흐름도, DFD) 순으로 수행.
- **Booch (부치) 방법**: 미시적/거시적 개발 프로세스 모두 사용.
- **Jacobson 방법**: Use Case를 강조.
- **Coad와 Yourdon 방법**: E-R 다이어그램을 사용하여 모델링.
- **Wirfs-Brock 방법**: 분석과 설계 간의 구분이 없음.

### 객체지향 설계 원칙 (SOLID)
- **SRP (단일 책임 원칙)**: 객체는 단 하나의 책임만 가져야 한다.
- **OCP (개방-폐쇄 원칙)**: 기존 코드를 변경하지 않고 기능을 추가할 수 있도록 확장에는 열려 있고 변경에는 닫혀 있어야 한다.
- **LSP (리스코프 치환 원칙)**: 자식 클래스는 최소한 부모 클래스에서 가능한 행위는 수행할 수 있어야 한다.
- **ISP (인터페이스 분리 원칙)**: 자신이 사용하지 않는 인터페이스와 의존 관계를 맺지 않아야 한다.
- **DIP (의존 역전 원칙)**: 추상성이 낮은 클래스보다 추상성이 높은 클래스와 의존 관계를 맺어야 한다.

## 6. UML (Unified Modeling Language)
- 시스템 개발 과정에서 의사소통이 원활하게 이루어지도록 표준화한 객체지향 모델링 언어.
- **UML의 관계**: 연관(Association), 집합(Aggregation), 포함(Composition, 변화에 영향), 일반화(Generalization, 상속), 의존(Dependency, 짧은 연관 유지), 실체화(Realization, 인터페이스 그룹화).
- **스테레오 타입**: 기본 기능 외에 추가 기능 표현. `<<include>>`, `<<extend>>`, `<<interface>>` 등 겹화살괄호(길러멧) 사용.

### 구조적(정적) 다이어그램
- **클래스 다이어그램**: 클래스와 속성, 관계 표현. (시스템 구조 파악)
- **객체 다이어그램**: 인스턴스를 특정 시점의 관계로 표현.
- **컴포넌트 다이어그램**: 구현 모듈인 컴포넌트 간 관계 및 인터페이스 표현.
- **배치 다이어그램**: 물리적 요소들의 위치와 통신 경로 표현.
- **복합체 구조 / 패키지 다이어그램**

### 행위적(동적) 다이어그램
- **유스케이스 다이어그램**: 사용자 관점에서 요구 분석. (시스템, 액터, 유스케이스, 관계로 구성).
- **순차(Sequence) 다이어그램**: 시간의 흐름에 따라 상호 작용하는 과정을 액터, 객체, 생명선(Lifeline), 메시지로 표현.
- **커뮤니케이션 다이어그램**: 메시지뿐만 아니라 객체 간의 연관까지 표현.
- **상태(State) 다이어그램**: 객체의 상태가 어떻게 변화하는지 표현. (럼바우 동적 모델링에 사용).
- **활동(Activity) 다이어그램**: 기능 처리 로직이나 조건에 따른 흐름 순서.

## 7. 소프트웨어 아키텍처 및 설계
- **상위 설계**: 아키텍처 설계, 예비 설계 (시스템 구조, DB, 인터페이스 설계).
- **하위 설계**: 모듈 설계, 상세 설계 (컴포넌트, 자료 구조, 알고리즘 설계).
- **설계 기본 원리**:
  - **단계적 분해 (Stepwise Refinement)**: 상위 개념부터 하위 개념으로 구체화하는 하향식 분할 기법.
  - **정보 은닉 (Information Hiding)**: 내부 절차를 감추어 다른 모듈이 접근하지 못하도록 함.
  - **모듈화 (Modularity)**: 기능들을 모듈 단위로 나누어 유지 관리 향상.
  - **추상화 (Abstraction)**: 포괄적 개념 설계 후 세분화.

### 아키텍처 패턴
- **파이프-필터 패턴 (Pipe-Filter)**: 데이터 스트림 절차의 각 단계를 필터로 캡슐화하고 파이프를 통해 전송. 재사용성이 좋고 확장이 용이 (예: UNIX의 쉘).
- **모델-뷰-컨트롤러 패턴 (MVC)**: 
  - **Model**: 핵심 기능과 데이터 보관.
  - **View**: 사용자에게 정보 표시.
  - **Controller**: 사용자 입력 요청 처리, 모델 제어.
- **기타 패턴**: 마스터-슬레이브(병렬 컴퓨팅), 브로커, 피어-투-피어, 이벤트-버스, 블랙보드(음성 인식 등), 인터프리터.

## 8. 모듈의 결합도와 응집도
- 소프트웨어 품질을 높이려면 **결합도는 약하게, 응집도는 강하게** 설계해야 한다. 팬인(Fan-In)은 높게, 팬아웃(Fan-Out)은 낮게 설계한다.

### 결합도 (Coupling) : 강함(나쁨) -> 약함(좋음)
- **내용 (Content)**: 한 모듈이 다른 모듈의 내부 기능/자료를 직접 참조. (가장 나쁨)
- **공통 (Common)**: 공유되는 공통 데이터 영역을 여러 모듈이 사용.
- **외부 (External)**: 외부의 다른 모듈에서 선언한 데이터를 참조.
- **제어 (Control)**: 논리 흐름을 제어하기 위해 제어 신호나 요소를 전달.
- **스탬프 (Stamp)**: 배열이나 레코드 등 자료 구조가 전달됨.
- **자료 (Data)**: 모듈 간 인터페이스가 자료 요소로만 구성. (가장 좋음)

### 응집도 (Cohesion) : 약함(나쁨) -> 강함(좋음)
- **우연적 (Coincidental)**: 서로 관련 없는 요소로만 구성. (가장 나쁨)
- **논리적 (Logical)**: 유사한 성격/형태로 분류되는 요소들이 모임.
- **시간적 (Temporal)**: 특정 시간에 처리되는 기능들을 모음.
- **절차적 (Procedural)**: 구성 요소들이 그 기능을 순차적으로 수행.
- **교환적/통신적 (Communication)**: 동일한 입력과 출력을 사용하여 서로 다른 기능을 수행.
- **순차적 (Sequential)**: 한 활동의 출력 데이터를 다음 활동의 입력 데이터로 사용.
- **기능적 (Functional)**: 모든 기능 요소들이 단일 문제와 연관되어 수행. (가장 좋음)

## 9. 디자인 패턴 (Design Pattern) - GoF
- 모듈의 세부적인 구현 방안을 설계할 때 참조할 수 있는 전형적인 해결 방식.
- **생성 패턴 (Creational)**: 객체의 생성과 참조 과정을 캡슐화.
  - 추상 팩토리, 빌더(건축하듯 조합), 팩토리 메소드(서브클래스에서 생성), 프로토타입(복제), 싱글톤(인스턴스 하나만 보장).
- **구조 패턴 (Structural)**: 클래스나 객체를 조합하여 더 큰 구조 생성.
  - 어댑터(호환성 변환), 브리지(기능과 구현 분리), 컴포지트(트리 구조, 복합/단일객체 구분 없음), 데코레이터(기능 덧붙임), 퍼싸드(상위 인터페이스/Wrapper 제공), 플라이웨이트(메모리 절약, 인스턴스 공유), 프록시(접근 제어/대행).
- **행위 패턴 (Behavioral)**: 상호작용이나 책임 분배 정의.
  - 책임 연쇄(고리를 따라 책임 넘어감), 커맨드(요청을 캡슐화/로그 기록), 인터프리터(문법 표현), 반복자(순차 접근), 중재자(상호작용 캡슐화로 결합도 감소), 메멘토(특정 시점 상태 복구, Undo), 옵서버(상태 변화 시 다른 객체에 전달), 상태, 전략(알고리즘을 캡슐화하여 교환), 템플릿 메소드(상위 골격/하위 구체화), 방문자(기능 분리 후 방문하여 수행).

## 10. 사용자 인터페이스 (UI) 및 기타
- **UI 설계 지침**: 사용자 중심, 사용성(우선 고려), 심미성, 오류 발생 해결, 직관성, 유효성, 학습성, 유연성.
- **UI 설계 도구**: 
  - **와이어프레임**: 초기에 레이아웃이나 뼈대 설계.
  - **목업(Mockup)**: 시각적으로 실제 화면과 유사하게 만든 정적 모형.
  - **스토리보드**: 와이어프레임에 설명, 흐름 추가. 디자이너/개발자의 최종 작업 지침서.
  - **프로토타입**: 인터랙션을 적용해 실제 구현된 것처럼 테스트 가능한 동적 모형.
- **코드(Code)의 종류**: 순차 코드(일련번호), 블록 코드(구분 코드), 10진 코드, 그룹 분류 코드, 연상 코드(명칭/약호), 표의 숫자 코드(물리적 수치 그대로, 유효 숫자), 합성 코드.
