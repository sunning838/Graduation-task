# 2과목 소프트웨어 개발

## 1. 자료 구조 (Data Structure)
- **개념**: 컴퓨터에 데이터를 저장하고 처리하는 방식을 체계적으로 구조화한 것이다. 데이터의 논리적, 물리적 관계에 따라 선형 구조와 비선형 구조로 나뉜다.

### 선형 구조 (Linear Structure)
- **연속 리스트 (Contiguous List, 배열)**: 연속되는 기억장소에 저장되는 자료 구조로 기억장소 이용 효율 밀도가 1로서 가장 좋다. 중간에 데이터를 삽입/삭제 시 빈 공간이 필요하여 자료의 이동이 수반된다.
- **연결 리스트 (Linked List)**: 자료들을 임의의 기억공간에 저장하되, 포인터(링크)를 이용하여 서로 연결시킨 자료 구조. 노드의 삽입/삭제가 용이하지만, 연결을 위한 포인터 공간이 추가로 필요하고 접근 속도가 상대적으로 느리다.
- **스택 (Stack)**: 리스트의 한쪽 끝으로만 자료의 삽입, 삭제 작업이 이루어지는 후입선출(LIFO; Last In First Out) 방식의 자료 구조. 함수 호출 순서 제어, 인터럽트 처리, 수식 계산 및 표기법 연산, 서브루틴 복귀 주소 저장 등에 응용된다. 가득 찬 상태에서 삽입 시 오버플로(Overflow), 빈 상태에서 삭제 시 언더플로(Underflow)가 발생한다.
- **큐 (Queue)**: 한쪽에서는 삽입, 다른 한쪽에서는 삭제가 이루어지는 선입선출(FIFO; First In First Out) 방식의 자료 구조. 시작과 끝을 표시하는 두 개의 포인터가 존재한다.

### 비선형 구조 (Non-Linear Structure)
- **트리 (Tree)**: 정점(Node)과 선분(Branch)을 이용하여 사이클을 이루지 않도록 구성한 그래프의 특수한 형태.
  - **관련 용어**: 근 노드(Root Node, 최상위 노드), 디그리(Degree, 특정 노드에서 뻗어 나온 가지의 수), 단말 노드(Terminal/Leaf Node, 자식이 하나도 없는 디그리가 0인 노드), 트리의 디그리(노드들의 디그리 중 가장 큰 수).
  - **트리의 운행법 (순회)**: 
    - Preorder(전위): Root → Left → Right
    - Inorder(중위): Left → Root → Right
    - Postorder(후위): Left → Right → Root
- **그래프 (Graph)**: 정점과 간선으로 구성된 자료구조. n개의 정점일 때 무방향 그래프의 최대 간선 수는 n(n-1)/2, 방향 그래프의 최대 간선 수는 n(n-1)이다.

### 수식의 표기법
- **전위 표기법(PreFix)**: 연산자 → Left → Right (예: +AB)
- **중위 표기법(InFix)**: Left → 연산자 → Right (예: A+B)
- **후위 표기법(PostFix)**: Left → Right → 연산자 (예: AB+)
- **변환 방식**: 연산의 우선순위에 따라 괄호를 묶은 뒤, 각 괄호의 연산자를 해당 괄호의 앞으로 빼면 Prefix, 뒤로 빼면 Postfix가 되며 최종적으로 괄호를 제거한다. 스택을 이용하여 처리한다.

## 2. 정렬 및 검색 알고리즘 (Sort & Search)
- **개념**: 대량의 데이터 집합에서 데이터를 일정한 기준에 따라 재배치(정렬)하거나, 원하는 데이터를 빠르게 찾아내는(검색) 알고리즘 기법이다.

### 정렬 알고리즘 (Sort)
- **삽입 정렬 (Insertion Sort)**: 두 번째 값부터 시작해 그 앞의 값들과 비교하여 알맞은 위치에 삽입하고 나머지를 뒤로 밀어내는 방법. 최악 시간 복잡도 O(n^2).
- **선택 정렬 (Selection Sort)**: 전체 값 중 최소값을 찾아 첫 번째 값과 교환하고, 그다음 최소값을 찾아 두 번째 값과 교환하는 과정을 반복. 최악 시간 복잡도 O(n^2).
- **버블 정렬 (Bubble Sort)**: 인접한 두 데이터를 비교하여 위치를 교환하며 정렬. 최악 시간 복잡도 O(n^2).
- **힙 정렬 (Heap Sort)**: 전이진 트리(Complete Binary Tree)를 이용한 정렬 방식. 평균 및 최악 시간 복잡도 모두 O(nlog2n)으로 효율적이다.
- **2-Way 합병 정렬 (Merge Sort)**: 이미 정렬되어 있는 두 개의 파일을 한 개의 파일로 합병하는 방식. 평균 및 최악 시간 복잡도 O(nlog2n).
- **퀵 정렬 (Quick Sort)**: 피벗(Pivot) 키를 기준으로 작은 값은 왼쪽, 큰 값은 오른쪽 서브파일로 분해시키는 분할 정복 방식. 평균 O(nlog2n), 최악 O(n^2).

### 검색 및 해싱 (Search & Hashing)
- **이분 검색 (Binary Search)**: 반드시 순서화(정렬)된 파일이어야 검색할 수 있으며, 중간 레코드 키 값과 비교하며 검색 대상을 절반씩 줄여나가는 방식. 중간 레코드 번호 M = (F+L)/2를 이용.
- **해싱 함수 (Hashing Function)**:
  - 제산법(Division): 레코드 키(K)를 해시표 크기보다 큰 가장 작은 소수(Q)로 나눈 나머지를 홈 주소로 삼음.
  - 제곱법(Mid-Square): 키 값을 제곱한 후 그 중간 부분의 값을 홈 주소로 삼음.
  - 폴딩법(Folding): 키 값을 여러 부분으로 나눈 후 더하거나 XOR(배타적 논리합)한 값을 홈 주소로 삼음.
  - 그 외 기수 변환법(Radix), 대수적 코딩법, 숫자 분석법, 무작위법 등이 있다.

## 3. 애플리케이션 테스트 (Application Test)
- **개념**: 애플리케이션에 잠재되어 있는 결함을 찾아내는 행위로, 소프트웨어가 고객의 요구사항을 만족하는지 확인(Validation)하고 명세서대로 정확히 수행되는지 검증(Verification)하는 절차이다.

### 테스트의 기본 원리
- **결함 집중 (Defect Clustering)**: 파레토 법칙에 따라 테스트로 발견된 오류의 80%는 20%의 특정 핵심 모듈에 집중된다는 원리.
- **살충제 패러독스 (Pesticide Paradox)**: 동일한 테스트 케이스를 지속적으로 반복 적용하면 벌레가 내성이 생기듯 더 이상 새로운 오류를 찾을 수 없는 현상.
- **오류-부재의 궤변 (Absence of Errors Fallacy)**: 소프트웨어의 결함을 모두 제거해도 사용자의 본래 요구사항을 만족시키지 못하면 품질이 높다고 볼 수 없다.
- **테스트 기반에 따른 테스트**: 명세 기반(요구사항 명세서 활용), 구조 기반(내부 논리 흐름 활용), 경험 기반(테스터의 직관이나 경험 활용)으로 나뉜다.

### 화이트박스 테스트 (White Box Test)
- 모듈의 원시 코드(Source Code)를 오픈시킨 상태에서 내부의 논리적인 모든 경로를 테스트하는 기법 (구조 기반 테스트).
- **종류**: 기초 경로 검사(Base Path Testing), 제어 구조 검사(조건 검사, 루프 검사, 데이터 흐름 검사).
- **검증 기준 (Coverage)**:
  - **문장 검증 (Statement)**: 소스 코드의 모든 구문이 한 번 이상 수행되도록 설계.
  - **분기 검증 (Branch/Decision)**: 소스 코드 내 모든 조건문의 참(True), 거짓(False) 경로가 한 번 이상 수행되도록 설계.
  - **조건 검증 (Condition)**: 조건문에 포함된 개별 조건식들의 결과가 참/거짓으로 한 번 이상 수행되도록 설계.
  - **분기/조건 기준**: 분기 검증과 조건 검증 기준을 모두 만족하는 통합 설계.

### 블랙박스 테스트 (Black Box Test)
- 프로그램의 내부 구조를 고려하지 않고 요구 명세를 기초로 기능이 완벽히 작동하는지 입증하는 테스트 (기능 테스트).
- **종류**: 
  - **동치 분할 검사(Equivalence Partitioning)**: 타당한 입력 값과 타당하지 않은 입력 값의 개수를 균등하게 분할하여 테스트.
  - **경계값 분석(Boundary Value Analysis)**: 동치 분할 기법을 보완하여, 오류 확률이 높은 입력 조건의 경계값을 테스트 케이스로 선정.
  - **원인-효과 그래프(Cause-Effect)**, **오류 예측 검사(Error Guessing)**, **비교 검사**.

### 개발 단계에 따른 테스트 (V-모델)
- **단위 테스트 (Unit Test)**: 최소 단위인 모듈/컴포넌트에 초점을 맞춰 코딩 직후 수행. 주로 구조 기반 테스트 시행.
- **통합 테스트 (Integration Test)**: 단위 테스트가 완료된 모듈들을 결합하여 하나의 시스템으로 완성시키는 과정의 테스트.
  - **하향식 (Top-Down)**: 상위 모듈에서 하위로. 하위 모듈 역할을 대신하는 시험용 가짜 모듈인 **스텁(Stub)**이 필요하다.
  - **상향식 (Bottom-Up)**: 하위 모듈에서 상위로. 상위 모듈 역할을 하는 제어 도구인 **드라이버(Driver)**가 필요하다.
  - **회귀 테스트 (Regression Test)**: 모듈 수정이나 통합 후 변경된 코드로 인해 새로운 결함이 생기지 않았음을 보증하기 위해 이전 테스트를 반복 수행.
- **시스템 테스트 (System Test)**: 실제 사용 환경과 유사하게 만든 환경에서 소프트웨어가 완벽히 수행되는지 점검.
- **인수 테스트 (Acceptance Test)**: 사용자 요구사항 충족 여부를 사용자가 직접 확인. 
  - **알파 테스트**: 개발자 장소에서 사용자가 개발자 통제하에 수행.
  - **베타 테스트(필드 테스팅)**: 실제 업무 환경에서 사용자가 직접 테스트하고 오류를 개발자에게 보고.

### 테스트 케이스, 오라클, 자동화 도구
- **테스트 케이스 (Test Case)**: 소프트웨어가 요구사항을 준수했는지 확인하기 위해 설계된 입력 값, 실행 조건, 기대 결과 등의 명세서.
- **테스트 오라클 (Test Oracle)**: 테스트 결과의 참/거짓 여부를 판단하기 위해 사전에 정의된 참 값을 대입하여 비교하는 기법.
  - 참 오라클(모든 오류 검출), 샘플링 오라클(특정 몇몇 값만), 추정 오라클, 일관성 검사 오라클.
- **테스트 자동화 도구**:
  - **정적 분석 도구**: 코드 실행 없이 코딩 표준, 스타일, 복잡도 분석 (pmd, cppcheck, SonarQube 등).
  - **동적 분석 도구**: 코드를 실행하여 메모리 누수, 스레드 결함 등을 확인 (Avalanche, Valgrind 등).

## 4. 소프트웨어 최적화 및 성능 분석
- **개념**: 애플리케이션의 성능을 정량적으로 파악하고, 불필요한 로직을 제거하여 최소한의 자원으로 신속하게 처리할 수 있도록 코드를 최적화하는 것이다.

### 애플리케이션 성능 지표
- **처리량 (Throughput)**: 일정 시간 내에 애플리케이션이 처리하는 일의 양.
- **응답 시간 (Response Time)**: 요청을 전달한 후 응답이 도착할 때까지 걸린 시간.
- **경과 시간 (Turn Around Time)**: 작업을 의뢰한 시간부터 처리가 완료될 때까지 걸린 시간.
- **자원 사용률 (Resource Usage)**: 작업 처리 동안의 CPU, 메모리, 네트워크 사용량.

### 알고리즘 복잡도
- **빅오 표기법 (Big-O)**: 최악의 실행 시간을 표기. (빠름) O(1) < O(log2n) < O(n) < O(nlog2n) < O(n^2) < O(2^n) (느림)
- **순환 복잡도 (Cyclomatic Complexity)**: 제어 흐름도 이론에 기초한 프로그램의 논리적 복잡도 척도.
  - 계산 공식: `V(G) = E(간선 수) - N(노드 수) + 2` 또는 제어 흐름도의 닫힌 영역 수와 일치.

### 소스 코드 최적화
- **클린 코드 작성 원칙**: 나쁜 코드(스파게티 코드, 외계인 코드)를 배제하고 가독성, 단순성, 의존성 배제, 중복성 최소화, 추상화를 적용하여 누구나 쉽게 수정할 수 있도록 작성하는 것.

## 5. 소프트웨어 패키징 및 형상 관리 (SCM)
- **개념**: 개발된 모듈들을 묶어 실행 가능한 배포용 설치 파일로 만드는 작업(패키징)과, 개발 과정 전반의 소스 코드 변경 사항을 추적 및 제어하는 체계적 활동(형상 관리)이다.

### 빌드 자동화 및 통합 개발 환경
- **IDE (통합 개발 환경)**: 코딩, 컴파일, 디버그, 배포 과정을 하나의 인터페이스로 통합한 도구.
- **빌드 자동화 도구**: 소스 코드를 실행 가능한 소프트웨어로 변환(컴파일)하고 테스트 배포를 자동화.
  - 종류: Ant, Maven, Gradle(안드로이드 환경에서 주로 사용, Groovy 기반), Jenkins(Java 기반, SVN/Git 연동 및 분산 빌드 특화).

### 제품 소프트웨어 패키징 및 배포
- 패키징은 개발자가 아닌 **사용자 중심으로** 진행되며, 구동에 필요한 최소 시스템 환경을 정의해야 한다.
- **릴리즈 노트 (Release Note)**: 개발 과정에서 정리된 기능 개선, 버그 수정 내역 등의 배포 정보를 최종 사용자에게 공유하는 문서. 현재 시제로 작성.
- **디지털 저작권 관리 (DRM)**: 디지털 콘텐츠가 불법적으로 사용되지 않도록 패키저, 클리어링 하우스(결제 및 라이선스 관리), 암호화 및 정책 관리 기술 등을 결합한 보안 메커니즘.

### 형상 관리 (SCM; Software Configuration Management)
- 소프트웨어 개발의 전 단계(유지보수 포함)에 적용되며, 변경 사항을 추적하고 무절제한 변경을 통제하여 가시성을 보장한다.
- **주요 기능**: 형상 식별(기준선/Baseline 설정), 버전 제어, 형상 통제(변경 요구 검토), 형상 감사(무결성 평가), 형상 기록.
- **주요 버전 관리 명령어**: 저장소(Repository), 체크아웃(소스 수정을 위해 받아오기), 체크인(수정 후 갱신), 커밋(Commit, 충돌 확인 후 최종 확정 반영).
- **형상 관리 도구 구조**:
  - **클라이언트/서버 방식 (CVS, SVN 등)**: 중앙 서버에서 버전 관리 수행. Subversion(SVN)은 trunk(메인 작업)와 branches(추가 작업) 디렉터리 구조를 갖는다.
  - **분산 저장소 방식 (Git 등)**: 로컬 저장소와 원격 저장소로 나뉘어 오프라인에서도 버전 관리가 가능. 스냅샷 방식으로 파일 변화를 기록.

## 6. 단위 모듈 통신 및 시스템 연계 기술
- **개념**: 서로 다른 모듈, 프로세스, 시스템 간에 데이터나 제어 흐름을 주고받기 위한 인터페이스와 네트워크 통합 아키텍처이다.

### IPC (Inter-Process Communication) 및 단위 모듈
- 모듈 및 프로세스 간 통신을 위한 대표적 메소드: Shared Memory, Socket(네트워크 소켓 이용), Semaphores(공유 자원 접근 제어), Pipes, Message Queueing.
- 단위 모듈 구현 시에는 독립적인 기능 동작을 위해 입력 데이터, 테스트 조건이 명세된 **테스트 케이스** 작성이 필수적이다.

### 시스템 연계 기술 및 미들웨어
- **미들웨어 (Middleware)**: 운영체제와 응용 프로그램 사이에서 다양한 서비스를 제공.
  - 종류: DB 미들웨어, RPC(원격 프로시저 호출), MOM(메시지 지향 비동기 통신), TP-Monitor(트랜잭션 처리 감시), ORB(객체 요청 브로커), WAS(웹 애플리케이션 서버).
- **EAI (Enterprise Application Integration)**: 기업 내 각종 플랫폼 간의 상호 연동을 위한 솔루션.
  - 토폴로지: Point-to-Point(1:1), Hub & Spoke(중앙 허브 병목 위험 존재), Message Bus(미들웨어 기반), Hybrid.
- **ESB (Enterprise Service Bus)**: 애플리케이션보다 **서비스 중심**의 통합을 지향하며, 결합도를 약하게 유지(Loosely Coupled)하는 표준 기반 인터페이스 솔루션.

### 인터페이스 데이터 포맷 및 보안
- **JSON**: 속성-값 쌍(Attribute-Value Pairs)으로 구성된 텍스트 기반 개방형 포맷. 비동기 처리인 AJAX에서 XML을 대체하여 널리 쓰임.
- **XML**: 사용자가 태그를 정의할 수 있는 다목적 마크업 언어.
- **인터페이스 보안**: 
  - 네트워크 영역: IPsec, SSL 등을 이용해 스니핑과 데이터 탈취 방지.
  - 애플리케이션/DB 영역: 시큐어 코딩 및 프로시저 암호화 적용.
- **데이터 무결성 검사 도구**: 시스템 침입에 의한 시스템 파일 변경 유무를 해시 함수로 감시 (Tripwire, AIDE 등).
